# OCR 第一阶段增强设计

**目标**

在不增加前端设置项、不改变数据库结构、不引入大模型的前提下，提升截图 OCR 在小字、浅色字、正文区域和文字稀疏场景下的识别稳定性，同时尽量控制额外耗电。

**范围**

- 保持现有对外接口 `extract_text(&Path) -> Result<Option<OcrResult>>` 不变
- 保持 [main.rs](/Users/wmy/data/Pycharm_Project/Work_Review/src-tauri/src/main.rs) 现有异步 OCR 调用结构不变
- 仅在 [ocr.rs](/Users/wmy/data/Pycharm_Project/Work_Review/src-tauri/src/ocr.rs) 内部增加自动策略

**非目标**

- 不新增前端开关或配置项
- 不新增数据库字段
- 不实现 Python 常驻 worker
- 不实现 C++ / ONNX 常驻版
- 不做按应用定制的区域模板

## 当前问题

当前 OCR 直接对整张图执行一次识别。对正文密集、字号正常的截图通常可用，但在以下场景容易漏字：

- 浏览器地址栏、窗口标题等顶部小字
- 编辑器、聊天工具中的浅色字和细字重
- 整图噪音较多，主体文字区域被背景元素稀释
- OCR 结果很短或清洗后大量缩水，实际是一次低质量识别

## 方案概述

第一阶段采用“整图优先，低质量才补救”的自动策略：

1. 先对原图执行一次默认 OCR
2. 对首轮结果做轻量质量评估
3. 只有命中低质量条件时，才触发区域裁剪和图像预处理
4. 对少量高价值区域执行补充 OCR
5. 合并并去重文本后返回最终结果

该方案的关键点是把额外计算成本限制在“少数识别较差的截图”上，而不是每张图都做多轮重处理。

## 组件设计

### 1. 策略入口

`extract_text` 由“直接调用引擎”调整为“策略入口”：

- `extract_text_once`：保留现有单次 OCR 能力，负责调用具体平台引擎
- `extract_text`：新增策略编排，负责首轮 OCR、质量评估、补救重试和结果合并

这样 [main.rs](/Users/wmy/data/Pycharm_Project/Work_Review/src-tauri/src/main.rs) 无需了解 OCR 内部细节。

### 2. 质量评估

首轮 OCR 结果满足任一条件时，判定为低质量：

- 无结果或清洗后文本为空
- 清洗后文本总长度过短
- 有效行数过少，且总字数明显不足
- 清洗后文本相对原始文本缩水过多
- 有效字符占比异常低，疑似以噪音符号为主

评估只依赖已有文本和简单规则，不引入额外模型。

### 3. 区域规划

低质量补救阶段仅扫描固定少量区域，全部使用相对比例裁剪：

- 顶部带：覆盖标题、标签页、导航和地址栏附近
- 中央主体区：覆盖文档正文、网页正文、聊天主体、编辑器主要内容
- 中上偏左主体区：覆盖常见桌面应用的主要文本区域

第一阶段不做应用级特化，避免引入额外维护成本。

### 4. 图像预处理

预处理只作用于补救区域，不对所有截图默认开启：

- 适度放大
- 灰度化
- 对比度增强
- 轻微锐化

第一阶段不做激进二值化，避免对 UI 截图产生过度破坏。

### 5. 结果合并

补救结果的文本只用于“补全”，不覆盖首轮主结果：

- 以首轮整图 OCR 文本为主
- 将补救区域中新增的文本行追加到结果中
- 统一执行清洗和去重
- 保留首轮 OCR 的文字框信息，避免裁剪重试后坐标失真

## 数据流

```text
原图路径
  -> 首轮整图 OCR
  -> 质量评估
     -> 质量合格：直接返回
     -> 质量较差：加载原图
        -> 规划高价值区域
        -> 对区域执行预处理
        -> 临时保存裁剪图
        -> 补充 OCR
        -> 合并文本并去重
        -> 返回最终结果
```

## 错误处理

- 首轮 OCR 失败时，允许进入补救流程，只要后续区域 OCR 能拿到文本即可
- 区域裁剪或预处理失败时，不中断整体流程，回退到首轮结果
- 任一补救区域 OCR 失败时，只忽略该区域，不影响其它区域
- 临时文件必须在每次补救后删除，避免残留

## 测试策略

第一阶段优先覆盖策略逻辑，不依赖真实系统 OCR 环境：

- 低质量判定规则
- 区域规划不越界
- 预处理输出尺寸和基础变换
- 文本合并去重
- 策略编排：好结果不重试，差结果触发补救并合并输出
